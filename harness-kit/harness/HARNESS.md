# Project Harness — <PROJECT NAME> (harness kit-0.1)

This document is the **meaning layer** of the harness: invariants,
authority, gates, and escalation signals. The other two layers live
elsewhere:

- **Facts** (state, history) → `harness/ledger.jsonl`, schema in
  `SCHEMA.md`. `STATE.md` is generated by `harness.py state`; never author
  it.
- **Mechanical rules** (checkable yes/no) → enforced by
  `harness.py validate`. Listed in `SCHEMA.md`; deliberately not restated
  here.

Tasks pin the harness version active when they start. Changing this
document is a `harness/process` decision: approver gate required, recorded
as a `decision` event plus a prose record in `decisions/process/`.

Sections marked **FILL IN** are the project-specific third of this
document. Answering them well is your project's Phase 0 — it is human work
and the most important customization in the kit. Each question names a
category where AI-driven development erodes systems; your answers are the
red lines agents will be held to. Worked examples from the kit's origin
project (Intelligence Hub) are included for calibration.

## Source Authority — FILL IN

List your sources of product truth in precedence order (most
implementation-specific accepted artifact wins). Example shape:

1. `contracts/` — frozen interface/schema contracts (if you have them)
2. `docs/architecture/` — architecture decisions
3. `docs/vision/` or README — rationale and terminology only; never

The harness governs process only. It must never become the primary source
of product truth.

## Architectural Invariants — FILL IN (the questionnaire)

Answer each category that applies. Skip ones that don't — but record the
skip. These change only with explicit approver approval plus a decision
record.

1. **Canonical interface.** What is the one interface everything goes
   through, and what is forbidden to bypass it?
   *(Worked example: "One shared Hub Core exposes one versioned Hub API;
   the desktop app always talks through the API and never absorbs core
   logic.")*
2. **Variant management.** What are your deployment/product variants, and
   what stops them becoming semantic forks?
   *(Example: "Desktop and Enterprise are profiles of the same core;
   differences are expressed through capability negotiation, never forked
   behavior.")*
3. **Trust boundaries.** What must never mix? Secrets, tenants, personal
   vs. organizational data?
   *(Example: "Personal and organizational secrets never mix.")*
4. **Egress.** What may never leave, or never leave to an unknown
   destination?
   *(Example: "No model request with an unknown destination.")*
5. **Sovereignty / reversibility.** What must always remain exportable,
   local-capable, or undoable?
   *(Example: "Offline capability for installed assets and full
   exportability are non-negotiable.")*
6. **Privilege separation.** What installation/permission boundaries must
   stay distinct?
   *(Example: "Artifact installation stays separate from product
   installation.")*
7. **Scope ratchet.** What is explicitly out of scope for the current
   version, however tempting?
   *(Example: "v1 run engines are X and Y only; marketplace and background
   engines are anticipated but out of scope.")*
8. **Conceptual core.** Which concept is the center of gravity whose
   semantics must not drift?
   *(Example: "Frames are the v1 center of gravity.")*

## Roles and Authority

**Approver (human, product owner).** Owns priorities, tradeoffs, phase
gates, escalations. Only the approver moves work to `done`, freezes
contracts, approves `architecture/contract` or `harness/process` decisions,
and promotes integration branches to stable.

**Planner.** Derives bounded task specs from the plan + architecture +
contracts: objective, dependencies, in/out of scope, verification and
acceptance criteria, dependency level, authoring mode, affected contracts.
Registers tasks via `task_created` events; commits each spec on the
integration branch before any task branch exists.

**Implementer.** Performs exactly one task spec on its task branch. May not
review, accept, merge, or mark its own work `machine_complete`/`done`.
`same-session-ok` authoring mode permits writing the spec and implementing
in one session — it never waives reviewer independence.

**Reviewer.** A different session/party from the implementer, always.
Judges the committed task-branch delta against the spec, contracts, and
changed artifacts — not against a newly invented preferred design. Records
one verdict with a Human Review Signal recommendation. On acceptance,
merges the task branch into the integration branch. Never merges to stable.

**Auditor.** A fresh session the approver runs at the human gate. Audits
the process record — never the product. Output is attention-direction,
never approval. See `roles/AUDITOR.md`.

**Facilitators.** Prepare approver decisions (phase setup/closeout); never
make them. Closeout before setup.

Roles are defined by responsibility, not by who fills them: any role may be
performed by a human, a machine, or a mixed loop. Record the acting party
with the optional `party` field so substitutions stay inspectable.

## Completion States and Gates

`not_started → in_progress → agent_review → machine_complete →
(human_review) → done`, with `blocked` reachable from any active state.
Legal transitions are validated mechanically.

`machine_complete` is machine acceptance, not final. `done` means the
approver accepted the work — only `done` is stable human-accepted meaning.
Dependency levels: `machine-level` downstream work may start on upstream
`machine_complete`; `human-level` waits for `done`. Default to
`human-level` for anything touching your invariants above.

**The approver is required for:** every transition to `done`; contract
freezes; PoC sign-offs; changes to invariants, phase sequencing, contracts,
or schemas; harness changes altering authority, evidence, or gates; any
decision with `human_gate: yes`.

**Human Review Signal.** Every verdict carries a recommendation:
`optional` (clean, batch it), `advised` (material note), `strong` (high
fan-out or boundary strain — advisory), `required` (contract-breaking or
escalate — hard-blocks the next task). Accumulation `N` (machine_complete
not yet done) also hard-blocks at 6+. Thresholds (2/4/6) are tunable per
project risk tolerance — change them in `harness.py` before data
collection, not after.

## Specs Are Reviewed Artifacts

Phase setup assigns each task an authoring mode: `separate` (a different
session/party must author the spec — high-fan-out, contract-shaping work)
or `same-session-ok` (routine, contract-fenced work). Every review runs a
Tier-1 spec check (well-formed, faithful to plan + contracts, no silent
scope narrowing). Tier-2 — independent re-derivation of proper scope — runs
when spec author == implementer or the task is `separate` mode.

## Issues

Observed problems become `issue` events first (`harness.py log issue ...`);
intake never authorizes a fix. Before fixing, route it (`issue_route`):
into the active task spec, a new task, a decision, an escalation, or an
explicit deferral with a named target. Capture and triage are decoupled —
bank the observation immediately, triage when convenient.

## Decisions

Prose records under `decisions/product/` (D-####) and `decisions/process/`
(P-####); each announced by a ledger `decision` event. Levels: `task-local`
(implementer may record when the spec delegates), `cross-task` (only when
explicitly delegated with bounds; reviewer must surface it),
`architecture/contract` and `harness/process` (approver approval before
proceeding, always `human_gate: yes`). Every record states authority,
authorization source, and escalation trigger.

## Escalation Signals — generic core + FILL IN

Stop and escalate (append an `escalation` event and stop) if work would:

- change or conflict with any invariant above
- introduce a hard-to-reverse framework, protocol, package, or storage
  decision not settled by the task
- change phase scope, or expand into named out-of-scope territory
- leave product behavior ambiguous in a way affecting user trust, security,
  or privacy
- make the harness the primary source of product truth

answers (each invariant usually implies one signal: "would bypass X,"
"would mix Y," "would weaken Z").

## PoC Gates — FILL IN

Name the 2–4 riskiest architectural assumptions and gate them with

- the **bootstrap loop** (does the core stand up end to end?)
- the **trust boundary** (do secrets/permissions actually separate?)
- the **core-semantics lifecycle** (does your central concept survive

## Git Discipline

- Branches: `task/<id>-<slug>` per task; one integration branch per phase
  (`phase/<id>-working`); stable branch (`main`) holds human-accepted work
  only.
- The task spec is committed on the integration branch *before* the task
  branch is created from that commit.
- Only the assigned implementer and reviewer commit to a task branch; the
  working tree rests on the integration branch between task work.
- Reviewer merges accepted tasks to integration; only the approver promotes
  integration to stable. Prefer merge commits.
- Post-merge, pre-`done` fixes reopen the loop on the task branch — and
  require a fresh re-review verdict (validated).

## The Loop (one task)

1. Planner: spec committed on integration branch → `task_created` event.
2. Implementer: task branch from the spec-baseline commit → `in_progress`
   → scoped work, contracts checked → commit → `agent_review` → stop.
3. Reviewer (fresh session): `harness.py brief <task> --role reviewer` →
   judge → judgment-only review record → `verdict` event → if accepted:
   merge to integration, `machine_complete`.
4. Approver: paced by the Human Review Signal → optionally
   `harness.py brief queue --role auditor` in a fresh session → read the
   audit → `human_accept` → `done`.
5. `harness.py validate` runs at every role handoff (and in the pre-commit
   hook); `harness.py state` regenerates the dashboard.
