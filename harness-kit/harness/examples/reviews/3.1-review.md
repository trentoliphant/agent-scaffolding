## What I judged

Spec is faithful to PLAN and the frozen hub-api contract (Tier 2 applied —
spec author and implementer shared a session under `same-session-ok`); no
scope narrowing. The implementation keeps export in Hub Core behind the
capability gate — the Desktop app only calls the API, so the profile
boundary holds. Filename uniqueness (ISSUE-0101) is fixed the right way:
content-addressed suffix, not a mutable counter.

## Risks worth a human's attention

- The export document embeds provider *names* (not keys) for attribution.
  That is contract-clean today but will constrain how provider renames work
  later. Acceptable; flagging it.
- Windows path normalization is untested (no Windows CI). Deferred to 3.4
  rather than guessed at — see the deferred route on the verdict event.

## Why `advised`, not `optional`

Export touches the sovereignty surface (exportability invariant), and one
deferred item lands in a future task. Nothing blocks; a human skim of the
export document format is worthwhile before downstream tasks rely on it.
